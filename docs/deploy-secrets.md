# Deploy Secrets Checklist

This checklist separates client-safe Flutter defines from backend-only secrets
for staging and production.

Use it as the source of truth before release, especially when wiring Supabase
Edge Functions, Stripe, Firebase, and auth redirects.

## Security Rules

- `SUPABASE_SERVICE_ROLE_KEY` must never go into Flutter client code.
- `STRIPE_SECRET_KEY` must never go into Flutter client code.
- `STRIPE_WEBHOOK_SECRET` must never go into Flutter client code.
- `FIREBASE_SERVICE_ACCOUNT_JSON` must never go into Flutter client code.
- `SUPABASE_ANON_KEY` can live in the client app.
- `STRIPE_PUBLISHABLE_KEY` can live in the client app.

| Variable                        | Where to set it                                      | Where it comes from                           | Example format                                       | Client-safe? | Environments                         |
| ------------------------------- | ---------------------------------------------------- | --------------------------------------------- | ---------------------------------------------------- | ------------ | ------------------------------------ |

| `STRIPE_PUBLISHABLE_KEY_TEST`   | Flutter `dart-define`                                | Stripe Dashboard -> Developers -> API keys    | `pk_test_...`                                        | Yes          | local, staging                       |
| `STRIPE_PUBLISHABLE_KEY`        | Flutter `dart-define`                                | Stripe Dashboard -> Developers -> API keys    | `pk_live_...`                                        | Yes          | production                           |

| `STRIPE_GOOGLE_PAY_TEST_ENV`    | Flutter `dart-define`                                | Internal project config                       | `true` or `false`                                    | Yes          | local, staging                       |

| `STRIPE_CONNECT_REFRESH_URL`    | Supabase Edge Function secret, CI secret             | Your hosted connect callback page             | `https://example.com/truffly/stripe/connect-refresh` | No           | staging, production                  |

## Flutter Build Recipes

### Local

```bash
flutter run \
  --dart-define=APP_ENV=local \
  --dart-define=AUTH_REDIRECT_BASE_URL=truffly://auth \
  --dart-define=SUPABASE_URL=http://10.0.2.2:54321 \
  --dart-define=SUPABASE_ANON_KEY=your-local-anon-key \
  --dart-define=STRIPE_PUBLISHABLE_KEY_TEST=pk_test_your_key
```

### Staging

```bash
flutter run \
  --dart-define=APP_ENV=staging \
  --dart-define=AUTH_REDIRECT_BASE_URL=truffly://auth \
  --dart-define=SUPABASE_URL=https://your-staging-project-ref.supabase.co \
  --dart-define=SUPABASE_ANON_KEY=your-staging-anon-key \
  --dart-define=STRIPE_PUBLISHABLE_KEY_TEST=pk_test_your_key
```

### Production Build

```bash
flutter build appbundle \
  --dart-define=APP_ENV=production \
  --dart-define=AUTH_REDIRECT_BASE_URL=truffly://auth \
  --dart-define=SUPABASE_URL=https://your-production-project-ref.supabase.co \
  --dart-define=SUPABASE_ANON_KEY=your-production-anon-key \
  --dart-define=STRIPE_PUBLISHABLE_KEY=pk_live_your_key
```

## Supabase CLI

Link the project first if needed:

```bash
npx supabase link --project-ref <project-ref>
```

List the currently configured secrets:

```bash
npx supabase secrets list
```

Set one or more secrets for the linked project:

```bash
npx supabase secrets set \
  SUPABASE_URL=https://your-project.supabase.co \
  SUPABASE_ANON_KEY=your_anon_key \
  SUPABASE_SERVICE_ROLE_KEY=your_service_role_key \
  STRIPE_SECRET_KEY=sk_live_or_sk_test \
  STRIPE_WEBHOOK_SECRET=whsec_your_webhook_secret \
  CRON_SECRET=your_cron_secret \
  FIREBASE_SERVICE_ACCOUNT_JSON='{"type":"service_account","project_id":"your-project-id"}' \
  PUSH_TEST_SECRET=your_staging_only_secret
```

Deploy one Edge Function:

```bash
npx supabase functions deploy publish_truffle
```

Deploy all functions you changed:

```bash
npx supabase functions deploy create_payment_intent
npx supabase functions deploy finalize_payment_attempt
npx supabase functions deploy publish_truffle
npx supabase functions deploy dispatch_notification_pushes --no-verify-jwt
npx supabase functions deploy send_test_push --no-verify-jwt
```

Push database migrations:

```bash
npx supabase db push
```

Functions that are triggered by cron or external webhooks are deployed with JWT
verification disabled and protected internally by their own secret checks:

- `cancel_unshipped_orders`
- `auto_complete_orders`
- `auto_create_missing_reviews`
- `retry_financial_operations`
- `stripe_webhook`
- `dispatch_notification_pushes`

`send_test_push` also runs with gateway JWT verification disabled because it
supports two explicit authorization paths inside the function:

- An authenticated user may send only to their own `user_id`.
- A staging operator may provide `x-push-test-secret: <PUSH_TEST_SECRET>`.

Example staging test:

```bash
curl -X POST \
  'https://<project-ref>.supabase.co/functions/v1/send_test_push' \
  -H 'Content-Type: application/json' \
  -H 'x-push-test-secret: <PUSH_TEST_SECRET>' \
  -d '{
    "user_id": "<staging-user-uuid>",
    "title": "Test Truffly",
    "body": "Questa è una notifica di test",
    "target_route": "/notifications",
    "type": "test_push"
  }'
```

The scheduled cron jobs send `Authorization: Bearer <cron-secret>`, and the
functions verify that header against `CRON_SECRET`.

## Supabase Auth URL Configuration

Set these values in Supabase Dashboard -> Authentication -> URL Configuration:

- Site URL: `http://truffly.framer.website`
- Additional Redirect URLs: `http://truffly.framer.website/**`
- Additional Redirect URLs: `truffly://**`

The Framer site is the web fallback only. Mobile auth links should use the
`truffly://` scheme and open the app directly.

Do not use localhost in staging or production auth redirects.

## Operational Notes

- `SUPABASE_ANON_KEY` and `STRIPE_PUBLISHABLE_KEY` are safe for the client app,
  but still need to be set correctly per environment.
- `SUPABASE_SERVICE_ROLE_KEY`, `STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET`, and
  `FIREBASE_SERVICE_ACCOUNT_JSON` must stay backend-only.
- `STRIPE_CONNECT_RETURN_URL` and `STRIPE_CONNECT_REFRESH_URL` are not secret,
  but they are deployment-specific and should be managed like backend config.

## Local Test-Only Variables

The Supabase function tests use separate `LOCAL_*` values. These are only for
test harnesses and local validation scripts, not for staging or production:

- `LOCAL_SUPABASE_URL`
- `LOCAL_SUPABASE_SERVICE_ROLE_KEY`
- `LOCAL_STRIPE_WEBHOOK_SECRET`
- `LOCAL_STRIPE_CONNECT_RETURN_URL`
- `LOCAL_STRIPE_CONNECT_REFRESH_URL`
- `LOCAL_PUBLISH_TRUFFLE_URL`
- `LOCAL_READY_STRIPE_ACCOUNT_ID`
